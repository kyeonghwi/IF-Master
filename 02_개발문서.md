## 4. 재처리(Retry) 시퀀스 다이어그램 및 멱등성 설계

### 4-1. 핵심 문제: 왜 멱등성이 필요한가

타임아웃으로 실패로 기록된 트랜잭션이 실제로는 외부 기관에서 처리 완료된 경우, 재처리를 그대로 실행하면 중복 처리(이중 출금, 이중 보고)가 발생합니다. 금융 시스템에서 이는 규제 위반 및 고객 피해로 이어집니다.

### 4-2. Idempotency Key 설계

```
idempotency_key = {대상기관}-{날짜}-{일련번호}
예: 금감원-20260419-00123
```

- 최초 호출 시 생성하여 DB에 저장
- 외부 기관에 요청 시 헤더(`Idempotency-Key`)로 함께 전송
- 재처리 시 동일 key로 외부 기관에 재요청 → 외부 기관은 동일 key의 중복 요청을 무시하고 이전 결과 반환

### 4-3. 재처리 시퀀스

```
담당자          Frontend        Backend         외부기관 API
  │                │               │                │
  │ [재처리] 클릭  │               │                │
  ├──────────────►│               │                │
  │               │ POST /retry/{id}               │
  │               ├──────────────►│                │
  │               │               │                │
  │               │               │ ① DB에서 idempotency_key 조회
  │               │               │                │
  │               │               │ ② 외부기관에 상태 조회 요청
  │               │               ├──────────────►│
  │               │               │◄──────────────┤
  │               │               │ (이미 처리됨?)  │
  │               │               │                │
  │               │               │ ③-A 이미 처리됨
  │               │               │   → DB 상태 SUCCESS로 업데이트
  │               │               │   → ALREADY_PROCESSED 반환
  │               │               │                │
  │               │               │ ③-B 미처리 확인
  │               │               │   → 동일 idempotency_key로 재전송
  │               │               ├──────────────►│
  │               │               │◄──────────────┤
  │               │               │                │
  │               │               │ ④ audit_log 기록 (실행자, 결과)
  │               │               │ ⑤ interface_log 업데이트
  │               │◄──────────────┤                │
  │◄──────────────┤               │                │
  │ 결과 표시      │               │                │
```

## 6. AI 활용 프로세스 (바이브코딩 실증)

본 섹션은 AI를 단순 코드 완성이 아닌 **설계 파트너**로 활용한 과정을 기록합니다.

### 6-1. 도메인 지식 습득

| 단계 | 활용 도구 | 활용 내용 |
|------|-----------|-----------|
| 금융 인터페이스 도메인 학습 | Gemini 3.1 Pro | "보험사 금융 IT 인터페이스에서 실제로 발생하는 장애 유형과 재처리 프로세스를 설명해줘" |
| 멱등성 설계 검토 | Claude Code | "타임아웃 실패 트랜잭션 재처리 시 멱등성을 보장하는 방법을 설계해줘" |
| Mock 데이터 생성 | Gemini 3.1 Pro | "보험사-금감원 간 SOAP 통신 실패 로그 JSON 20건 생성해줘" |

### 6-2. 코드 구현

| 단계 | 활용 도구 | 활용 내용 |
|------|-----------|-----------|
| FastAPI 재처리 로직 | Antigravity | Idempotency Key 조회 → 외부 상태 확인 → 재전송 파이프라인 자동 생성 |
| Next.js 대시보드 UI | Claude Code | 다크모드 금융 관제 대시보드 컴포넌트 구조 설계 및 구현 |
| PostgreSQL 스키마 | Antigravity | ERD 기반 DDL 자동 생성 및 인덱스 최적화 |
| Interface Registry | Claude Code | CRUD + 온디맨드 실행 API 설계 및 구현, 멱등성 키 자동 생성 로직 |
| 성능 관리 API | Claude Code | PERCENTILE_CONT 기반 P95/P99 집계 쿼리 설계 및 SLA 준수율 계산 |
| Mock 프로토콜 분기 | Claude Code | REST/SOAP/MQ/BATCH/SFTP 각 특성(지연·성공률)에 맞는 시뮬레이션 로직 |
| 성능 관리 대시보드 | Claude Code | P95 막대 그래프, SLA KPI 카드, 느린 인터페이스 알림 UI 구현 |

### 6-3. 검증 및 개선

| 단계 | 활용 도구 | 활용 내용 |

| 코드 리뷰 | Claude Code | 멱등성 로직 엣지케이스 검토 ("동시에 두 명이 재처리 클릭하면?") |

## 7. 기술 스택 선택 근거

| 기술 | 선택 이유 | 대안 대비 장점 |
|------|-----------|----------------|
| FastAPI (Python) | 금융 도메인 선호 언어, 비동기 처리, 자동 API 문서 생성 | Spring Boot 대비 프로토타입 구축 속도 3배 이상 빠름 |
| Next.js | AI 코드 생성 최적화, SSR + CSR 혼합 가능 | CRA 대비 배포 및 성능 최적화 용이 |
| PostgreSQL | ACID 트랜잭션 보장, 금융 시스템 표준 RDBMS | SQLite 대비 동시성·신뢰성 우위 |
| Render | FastAPI 컨테이너 무료 배포, PostgreSQL 포함 | Heroku 대비 무료 플랜 제공 |
