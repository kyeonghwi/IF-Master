## 3. 핵심 설계 결정

### 3-1. 멱등성 재처리 설계

**문제**: 타임아웃으로 실패 기록이 남은 트랜잭션이 실제로는 외부 기관에서 처리 완료된 경우, 재처리 시 이중 출금·이중 보고가 발생한다. 금융 규제 위반이다.

**해결**: 재처리 전 Idempotency Key로 외부 기관의 처리 여부를 먼저 조회한다.

```
Idempotency Key 형식: {대상기관[:6]}-{yyyyMMdd}-{uuid8}
예: 금감원-20260420-a3f7c291
```

**재처리 흐름**:

```
재처리 요청 (POST /api/retry/{id})
    │
    ├─ ① 권한 확인 (ADMIN | OPERATOR)
    │
    ├─ ② SELECT ... FOR UPDATE  ← 동시 재처리 Race Condition 방지
    │
    ├─ ③ 상태 확인 (FAILED | PENDING만 허용, 최대 3회 이내)
    │
    ├─ ④ AuditLog INSERT (RETRY_REQUEST)
    │
    ├─ ⑤ GET /mock/status?key={idempotency_key}
    │       ├─ Case A: 이미 처리됨
    │       │     → log.status = SUCCESS 업데이트
    │       │     → AuditLog (RETRY_SKIP)
    │       │     → "이미 처리됨 (멱등성 보장)" 반환
    │       │
    │       └─ Case B: 미처리 확인
    │             → POST /mock/call?key={key}  (동일 Key 재전송)
    │             → log.status 업데이트
    │             → AuditLog (RETRY_SUCCESS | RETRY_FAILED)
    │
    ├─ ⑥ COMMIT
    │
    └─ ⑦ SSE broadcast → 연결된 모든 클라이언트
```

### 3-2. 온디맨드 실행 — 내부 함수 직접 호출

**초안**: `POST /interfaces/{id}/execute` → `httpx`로 `/mock/call` 루프백 호출

**문제**: 같은 프로세스 내 루프백 HTTP 호출은 소켓 오버헤드 + 추적 어려움

**결정**: `mock_service.dispatch()` 함수를 직접 호출

```python
outcome, response_ms, response_json = await mock_service.dispatch(
    key=idempotency_key, protocol=cfg.protocol, timeout_ms=cfg.timeout_ms, db=db
)
```

### 3-3. SSE vs. WebSocket

**선택**: SSE (Server-Sent Events)

재처리 결과·실행 결과를 클라이언트에 push하는 단방향 스트림으로 충분하다. WebSocket은 양방향이 필요할 때 사용한다. SSE는 HTTP 위에서 동작해 Vercel/Render 프록시 통과가 쉽고, 자동 재연결이 브라우저 표준으로 내장돼 있다.

**이벤트 타입**:

| 이벤트 | 발생 시점 | 데이터 |
|--------|---------|--------|
| `new_tx` | APScheduler 30초 자동 생성 | `{id, status, service_name, ...}` |
| `retry_result` | 재처리 완료 | `{id, result, message}` |
| `execution_result` | 온디맨드 실행 완료 | `{id, result, service_name, response_ms}` |
| `ping` | 30초 keepalive | `{}` |

### 3-4. 성능 집계 — PostgreSQL PERCENTILE_CONT

P95/P99를 애플리케이션에서 계산하면 전체 response_ms를 메모리에 올려야 한다. PostgreSQL `PERCENTILE_CONT(0.95) WITHIN GROUP (ORDER BY response_ms)` 집계 함수로 서버 내에서 처리한다. 행이 수만 건이어도 단일 쿼리로 해결.

SLA 기준: `response_ms < 3,000ms`

### 3-5. 인증 — JWT + httpOnly Cookie

`localStorage`에 토큰 저장 시 XSS로 탈취 가능하다. httpOnly Cookie는 JavaScript에서 접근 불가 → XSS 방어. SameSite=Lax로 CSRF 위험 완화. Vercel(프론트)과 Render(백엔드) 도메인이 달라 `SameSite=None; Secure` 적용.

## 6. 기술 스택 선택 근거

### Python / FastAPI vs. Spring Boot

| 기준 | FastAPI | Spring Boot |
|------|---------|-------------|
| 프로토타입 속도 | 빠름 (타입힌트 → 자동 스키마) | 느림 (설정·보일러플레이트) |
| 비동기 지원 | 네이티브 (async/await) | WebFlux 필요 |
| AI 코드 생성 품질 | Claude/Gemini 최적화 | 구조 복잡해 품질 편차 |

**결론**: 프로토타입 + 바이브코딩 환경에서 FastAPI가 3배 이상 빠르다.

### PostgreSQL vs. SQLite

APScheduler 30초 자동 생성 + 사용자 재처리 + SSE 브로드캐스트가 동시에 발생하는 구조에서 SQLite는 테이블 락 문제가 있다. PostgreSQL MVCC가 이를 해결한다.

### 모노레포 구조

포트폴리오 프로젝트는 한 레포에서 전체 코드를 보여주는 게 심사자 편의에 맞다. 배포는 Vercel(프론트)/Render(백엔드)로 독립적이므로 단점이 없다.

## 9. 구현하지 않은 것과 이유

| 항목 | 이유 |
|------|------|
| Alembic 마이그레이션 | `create_all` 자동 처리로 충분. 프로토타입 오버헤드 불필요 |
| Redis 캐시 | 현재 트래픽 규모에서 불필요 |
| E2E 테스트 | API 수준 동작 검증으로 충분 |
| 실제 외부 기관 연동 | 포트폴리오 범위 초과, Mock 서버로 대체 |
| PII 마스킹 구현 | 시드 데이터에 실제 개인정보 없음. 설계 명세만 문서화 |
