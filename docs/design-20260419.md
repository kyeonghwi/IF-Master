## Problem Statement

채용 포트폴리오용 보험사 금융 인터페이스 통합 관제 시스템. 심사자가 30초 안에 핵심 기능(실시간 모니터링 + 원클릭 재처리 + 멱등성)을 체험하고 감동받아야 한다. 빈 대시보드, 콜드 스타트, 로그인 장벽이 데모 실패의 3대 원인이다.

## What Makes This Cool

- **실시간 트랜잭션 스트림**: 대시보드가 스스로 살아서 움직인다 (30초마다 새 트랜잭션 자동 생성)
- **금융 도메인 지식의 코드화**: "멱등성"을 단순 설명이 아닌 실제 동작하는 시퀀스로 보여준다
- **원클릭 데모 접근**: 심사자가 버튼 하나로 바로 핵심 기능 체험

## Constraints

- 스택: FastAPI (Python) + Next.js + PostgreSQL
- 배포: Vercel (프론트) + Render 단일 서비스 (백엔드 + Mock 서버 내장)
- 무료 플랜 사용 (Render 무료 PostgreSQL 포함)
- AI 도구: Claude Code, Antigravity, Gemini 3.1 Pro

## Premises (합의된 전제)

1. 원클릭 재처리 + 멱등성 보장이 이 포트폴리오의 핵심 차별점이다
2. Render 무료 플랜 콜드 스타트(30초)가 심사자 경험을 망친다 → GitHub Actions ping으로 해결
3. DB에 미리 로드된 Mock 데이터가 없으면 데모가 비어 보인다
4. 심사자가 로그인 없이 바로 핵심 기능을 볼 수 있어야 한다 (데모 원클릭 로그인)
5. Mock 서버를 FastAPI 메인 앱에 내장 → Render 서비스 1개로 유지

## Approaches Considered

### Approach A: MVP 우선
Mock 서버 내장 + 시드 데이터 + 데모 계정 하드코딩. Effort: S (2-3일). 실시간성 없음.

### Approach B: 설계 충실 구현
개발문서 1:1 구현 + JWT 완전 구현 + Mock 서버 독립 배포. Effort: M (5-7일). Render 콜드 스타트 2배.

### Approach C: 데모 임팩트 최대화 (선택)
실시간 스트림 + 자동 트랜잭션 생성 + Render 핀 + 데모 로그인. Effort: M (4-5일).

## Recommended Approach

**Approach C: 데모 임팩트 최대화**

### 핵심 추가 사항 (기존 개발문서 대비)

| 항목 | 기존 계획 | 변경/추가 |
|------|-----------|-----------|
| Mock 서버 | 독립 Render 서비스 | FastAPI 내장 라우터(`/mock/*`) |
| 데이터 생성 | 시드 스크립트 1회 실행 | APScheduler 30초 주기 자동 생성 |
| 실시간 대시보드 | 폴링 방식 | SSE (Server-Sent Events) |
| 인증 | JWT 완전 구현 | JWT 구현 + 데모 원클릭 버튼(test_admin 자동 로그인) |
| Render 콜드 스타트 | 미대응 | GitHub Actions 5분 cron ping |

### 구현 순서 (빌드 시퀀스, 총 ~23시간)

```
Step 1: DB 스키마 + 시드 스크립트 (0.5h)
  - interface_log, audit_log 테이블 생성 (개발문서 ERD 그대로)
  - idempotency_key 컬럼: interface_log.idempotency_key UNIQUE VARCHAR(64)
    → 키 형식: {대상기관}-{yyyyMMdd}-{UUID 앞 8자리}
    → 재처리 시 동일 키로 Mock 서버 호출 (Mock은 동일 키면 이전 결과 반환)
  - 시드: 500건 Mock 데이터 (Gemini 3.1 Pro 활용, status 비율: SUCCESS 70% / FAILED 20% / PENDING 10%)

Step 2: FastAPI 백엔드 코어 (8h)
  2-1. /api/stats, /api/logs, /api/logs/{id} → 개발문서 명세 그대로
  2-2. /api/retry/{id} → 멱등성 로직
       전제: FAILED 또는 PENDING 상태가 아닌 경우 즉시 400 반환 ("재처리 불가 상태")
       ① DB에서 idempotency_key 조회
       ② GET /mock/status?key={idempotency_key} 호출 (Mock 상태 확인)
       ③-A 이미 처리됨 → interface_log.status = SUCCESS, audit_log(action=RETRY_SKIP, result=ALREADY_PROCESSED) 기록
       ③-B 미처리 → POST /mock/call?key={idempotency_key} 호출 (최대 5초 대기, 지연 응답도 성공/실패 결과로 처리)
             성공 → interface_log.status = SUCCESS, audit_log(action=RETRY_SUCCESS, result=SUCCESS)
             실패 → interface_log.status = FAILED, interface_log.retry_count += 1, audit_log(action=RETRY_FAILED, result=FAILED)
       ④ SSE 브로드캐스트: "event: retry_result" (모든 연결된 클라이언트에 전송, 단일 관리자 데모 가정)
  2-3. /mock/* 라우터 (FastAPI 내장, 별도 router 파일)
       - GET /mock/status?key={key}: key 존재 시 이전 결과 반환, 없으면 "not_found"
       - POST /mock/call?key={key}: 성공 60% / 실패 30% / 지연(3초) 10% 랜덤 응답
         실패 응답: {"status": "FAILED", "error": "Connection timeout", "code": 503}
         성공 응답: {"status": "SUCCESS", "data": {"processed_at": "<timestamp>"}}
  2-4. APScheduler (BackgroundScheduler, in-memory, fire-and-forget)
       - 30초마다 랜덤 트랜잭션 1-3건 interface_log에 INSERT
       - Render 재시작 시 스케줄러도 재시작 → 데이터는 DB에 영속, 스케줄러 상태는 재시작으로 리셋 (허용)
       - APScheduler 콜백 → asyncio.Queue 또는 queue.Queue + asyncio.run_coroutine_threadsafe로 SSE generator에 이벤트 전달
  2-5. SSE 엔드포인트: GET /api/stream
       - EventSourceResponse (sse-starlette 라이브러리)
       - Render 100초 타임아웃 이전에 매 30초마다 ping 이벤트 전송 ("event: ping\ndata: {}\n\n")
       - 신규 트랜잭션 생성 시 "event: transaction\ndata: {id, status, target_org, called_at}\n\n"
       - 재처리 완료 시 "event: retry_result\ndata: {id, result, message}\n\n"

Step 3: JWT 인증 (2h)
  - POST /auth/login → JWT 발급 (HS256, 24시간 만료)
  - 데모 계정: test_admin / demo1234 환경변수 또는 하드코딩
  - 프론트에 "데모로 바로 체험하기" 버튼 → test_admin 자동 POST /auth/login → localStorage 저장

Step 4: Next.js 프론트엔드 (10h)
  - 다크모드 금융 관제 대시보드 (Claude Code 활용)
  - UI 라이브러리: TanStack Table (그리드), Recharts (차트), shadcn/ui (Toast/Button/Badge)
  - 상단: 요약 지표 카드 (총/성공/실패/지연) — FAILED=bg-red-500, SUCCESS=bg-green-500, PENDING=bg-yellow-500
  - 중단: Recharts 시계열 그래프 (EventSource로 transaction 이벤트 수신 → 차트 실시간 갱신)
    SSE 연결 전: "연결 중..." 스피너 표시 (빈 차트 방지)
  - 하단: TanStack Table 로그 그리드
    기본 정렬: status=FAILED 먼저, 그 다음 called_at DESC (심사자가 즉시 FAILED 행 발견)
    상태 배지: FAILED=red, SUCCESS=green, PENDING=yellow
  - 슬라이드 패널: fixed position overlay (오른쪽 40%, 차트 리플로우 없음)
    JSON/XML 전문 탭 + Audit Log 탭
  - 재처리 버튼:
    클릭 시 → loading spinner + disabled (Mock 호출 최대 5초 동안)
    결과 → Toast (5초 유지 + X 닫기 버튼, 충분히 읽을 시간 확보)
    Case A: "이미 처리됨 (멱등성 보장)" Toast
    Case B: "재처리 성공" Toast
  - EventSource 자동 재연결: onError 시 3초 후 재연결 + "SSE 재연결 중..." UI 표시
  - 데모 원클릭 로그인 버튼 → POST /auth/login → httpOnly 쿠키 저장

Step 5: 배포 + 콜드 스타트 방지 (2h)
  - Vercel: Next.js 자동 배포 (main 브랜치 push 시 자동, 별도 deploy.yml 불필요)
  - Render: FastAPI Dockerfile 배포 (main 브랜치 push 시 자동, 별도 deploy.yml 불필요)
  - GitHub Actions: .github/workflows/keep-alive.yml 만 추가
    (5분 cron: curl GET https://{render-url}/health)
    ※ GitHub Actions 무료 계정: 60일 이상 커밋 없으면 cron 비활성화 → 제출 전 커밋으로 유지
```

### Audit Log 기록 시점

```
트리거                          action          result
재처리 요청 시작                RETRY_REQUEST   PENDING
멱등성: 이미 처리됨             RETRY_SKIP      ALREADY_PROCESSED
재처리 성공                     RETRY_SUCCESS   SUCCESS
재처리 실패 (포함 지연 후 실패) RETRY_FAILED    FAILED
```
※ Mock 지연 응답(10%): 최대 5초 대기 후 성공/실패로 정상 처리 (타임아웃 액션 없음)

### 멱등성 데모 시나리오 (심사자 체험 플로우)

```
1. 대시보드 접속 → 실시간으로 트랜잭션 자동 생성 중 (살아있는 느낌)
2. 실패 트랜잭션 클릭 → 우측 슬라이드: Request/Response JSON 확인
3. [재처리] 클릭
   - Case A: 외부기관이 이미 처리 → Toast "이미 처리됨 (멱등성 보장)"
   - Case B: 미처리 확인 → 재전송 성공 → Toast "재처리 성공"
4. Audit Log 탭에서 실행자/시각 확인
```

## Engineering Fixes (autoplan Phase 3 AUTO-FIX)

### ERD 추가: mock_responses 테이블
```
mock_responses
  id              UUID      PK
  idempotency_key VARCHAR(64) UNIQUE NOT NULL
  status          VARCHAR(10)  -- SUCCESS or FAILED
  response_json   TEXT
  created_at      TIMESTAMP DEFAULT NOW()
```
→ `/mock/status?key=` 는 이 테이블 조회. Render 재시작해도 Mock 상태 유지.

### SSE 멀티클라이언트 패턴
```python
# FastAPI lifespan
client_queues: Set[asyncio.Queue] = set()

# APScheduler callback (다른 스레드)
def on_new_transaction(loop, event):
    for q in client_queues:
        asyncio.run_coroutine_threadsafe(q.put(event), loop)

# SSE generator
async def stream():
    q = asyncio.Queue()
    client_queues.add(q)
    try:
        while True:
            event = await asyncio.wait_for(q.get(), timeout=30)
            yield event
    finally:
        client_queues.discard(q)
```

## Security & Technical Fixes (autoplan Phase 1 AUTO-FIX)

1. **CORS**: FastAPI에 CORSMiddleware 추가 (origin=Vercel URL, credentials=True)
2. **JWT 저장소**: localStorage → httpOnly 쿠키 (XSS 방지). POST /auth/login 응답에 Set-Cookie
3. **JWT_SECRET**: 환경변수로 관리 (절대 하드코딩 금지). Render 환경변수 설정 필수
4. **Max retry cap**: POST /api/retry/{id}에서 `retry_count >= 3`이면 400 반환
5. **Error handling**: retry 엔드포인트에 명시적 예외 처리 (404/400/httpx.TimeoutException/SQLAlchemyError)
6. **README**: GIF 또는 30초 데모 영상 링크 포함 (GitHub 첫 스크롤에서 임팩트)

## What I noticed about how you think

- "채용 포트폴리오 / 데모" 선택 직후 모든 전제에 동의 → 심사자 관점에서 이미 생각하고 있었음. 콜드 스타트나 빈 대시보드 문제를 지적받기 전에 이미 인식하고 있었을 가능성이 높다.
- Render 콜드 스타트 문제를 "마지막에 해결하자"가 아닌 "모두 동의"로 선택 → 데모 품질에 대한 감각이 있다. 완성도를 타협하지 않으려는 태도.
- Approach C(가장 복잡하지만 임팩트 최대)를 선택 → 빠르게 끝내는 것보다 잘 끝내는 것을 선호한다.
