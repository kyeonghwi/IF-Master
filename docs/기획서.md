## 2. 추진 배경 및 핵심 문제

보험사는 금융감독원, 결제대행사, 제휴 보험대리점 등 수십 개 외부 기관과 REST, SOAP, MQ, Batch, SFTP 등 이종 프로토콜로 연동된다. 현행 운영 방식에는 두 가지 핵심 문제가 있다.

### 문제 1: 가시성 부재

장애 발생 시 각 시스템의 로그를 개별적으로 확인해야 한다. 어떤 인터페이스가 지금 살아있는지, 어디서 실패했는지를 한 화면에서 볼 수 없어 MTTR(평균 복구 시간)이 길어진다.

### 문제 2: 안전한 재처리 불가

타임아웃으로 실패 기록이 남은 트랜잭션을 재처리할 때, 외부 기관이 이미 처리를 완료한 경우 중복 실행(이중 출금, 이중 보고)이 발생한다. 금융 시스템에서 이는 규제 위반 및 고객 피해로 이어진다.

---

## 3. 개발 목표

이종 프로토콜 간의 연동 상태를 **단일 화면(Single Pane of Glass)** 에서 실시간 모니터링하고, 실패한 트랜잭션을 **멱등성(Idempotency)을 보장하며** 즉각 재처리할 수 있는 통합 관리 웹앱 프로토타입 구축.

## 4. 핵심 기능 정의

### 기능 1. 통합 모니터링 대시보드 (Control Tower)

- 전체 인터페이스 처리 현황(성공/실패/지연) 실시간 시각화 — 도넛 차트, 시계열 그래프
- 기관별(금감원, 은행 등) · 프로토콜별(REST, Batch 등) 필터링 조회
- APScheduler가 30초마다 트랜잭션을 자동 생성하여 실시간 데이터 유지
- SSE(Server-Sent Events)로 페이지 새로고침 없이 화면 자동 갱신

### 기능 2. 통합 로그 관리 및 상세 조회

- 분산된 인터페이스 호출 로그를 하나의 테이블에서 통합 조회
- 상태·프로토콜·기관·기간 필터 조합 지원
- 행 클릭 시 슬라이드 패널: Request/Response Payload 전문(JSON/XML) + Stack Trace

### 기능 3. 원클릭 재처리 — 멱등성 보장 (핵심 비즈니스 로직)

- FAILED 또는 PENDING 상태 트랜잭션에 대해 담당자가 대시보드에서 즉시 재요청 실행
- **멱등성 보장**: 재처리 전 외부 기관의 실제 처리 여부를 Idempotency Key로 먼저 조회하여 중복 처리 원천 차단
- `SELECT FOR UPDATE`로 동시 재처리 Race Condition 방지
- 재처리 실행자 · 시각 · 결과를 Audit Log로 전수 기록 → 금융 감사 요건 충족
- 최대 3회 재처리 초과 시 `RETRY_LIMIT_EXCEEDED` 에러 반환

### 기능 4. Interface Registry + 온디맨드 실행

- 인터페이스 등록(POST) · 수정(PUT) · 삭제(DELETE) · 활성/비활성 토글(PATCH)
- 등록된 인터페이스를 즉시 테스트 실행 — 결과가 SSE로 연결된 모든 클라이언트에 브로드캐스트

### 기능 5. 성능 관리 (P95/SLA)

- 기간 · 프로토콜 · 기관 필터 조합으로 집계 범위 변경
- PostgreSQL `PERCENTILE_CONT`로 P95/P99 응답시간 서버 내 집계
- SLA 기준(3,000ms): 준수율 및 초과 인터페이스 알림

### 기능 6. Mock 인터페이스 서버

실제 금융망 연결 없이 장애 시나리오 재현. 프로토콜별 특성을 반영한 응답 시뮬레이션:

| 프로토콜 | 응답 범위 | 성공률 | 근거 |
|----------|-----------|--------|------|
| REST | 200~1,500ms | 70% | HTTP REST API 표준 |
| SOAP | 500~3,000ms | 65% | XML 파싱 + 레거시 시스템 |
| MQ | 100~500ms | 85% | 큐 기반 비동기 ACK |
| BATCH | 5,000~30,000ms | 80% | 대량 데이터 처리 지연 |
| SFTP | 1,000~5,000ms | 75% | 파일 전송 오버헤드 |

## 6. AI 바이브코딩 활용 계획

본 프로젝트는 기획-개발-검증 전 과정에 AI를 **설계 파트너**로 활용하는 바이브코딩 방법론을 적용한다.

| 단계 | 활용 도구 | 활용 내용 |
|------|-----------|-----------|
| 도메인 지식 습득 | Gemini 3.1 Pro | "보험사 금융 IT 인터페이스에서 실제로 발생하는 장애 유형과 재처리 프로세스를 설명해줘" |
| 멱등성 설계 검토 | Claude Code | "타임아웃 실패 트랜잭션 재처리 시 멱등성을 보장하는 방법을 설계해줘" |
| Mock 데이터 생성 | Gemini 3.1 Pro | "보험사-금감원 간 SOAP 통신 실패 로그 JSON 20건 생성해줘" |
| FastAPI 재처리 로직 | Antigravity | Idempotency Key 조회 → 외부 상태 확인 → 재전송 파이프라인 자동 생성 |
| Next.js 대시보드 UI | Claude Code | 다크모드 금융 관제 대시보드 컴포넌트 구조 설계 및 구현 |
| PostgreSQL 스키마 | Antigravity | ERD 기반 DDL 자동 생성 및 인덱스 최적화 |
| 코드 리뷰 | Claude Code | 멱등성 로직 엣지 케이스 검토 ("동시에 두 명이 재처리 클릭하면?") |

## 7. 화면 설계 (UI 구조)

```
┌────────────────────────────────────────────────────┐
│ (상단) 요약 지표 카드                               │
│  총 트랜잭션 N건 | 성공률 N% | 🔴 장애 N건          │
├────────────────────────────────────────────────────┤
│ (중단) 시계열 그래프                                │
│  시간대별 프로토콜(REST/SOAP/BATCH…) 처리량·에러   │
├────────────────────────────────────────────────────┤
│ (하단) 인터페이스 로그 테이블                       │
│  발생일시 | 기관 | 프로토콜 | 서비스명 | 상태 | [재처리] │
│                                                    │
│  → 행 클릭 시 우측 슬라이드 패널                   │
│     Request/Response JSON/XML 전문                 │
│     + Audit Log (재처리 이력)                      │
└────────────────────────────────────────────────────┘
```

4개 화면 구성: Control Tower (모니터링) · Interface Logs (로그) · IF Registry (등록·실행) · Performance (성능 관리)

## 9. 기능 수용 기준 (Acceptance Criteria)

### 통합 모니터링
- 로그인 후 최근 24시간 트랜잭션 통계가 즉시 표시된다
- APScheduler 30초 자동 생성 데이터가 새로고침 없이 화면에 반영된다
- 프로토콜 · 기관 필터 적용 시 해당 조건 데이터만 표시된다

### 재처리 (멱등성)
- FAILED · PENDING 상태에만 재처리 버튼이 활성화된다
- 외부 기관이 이미 처리한 경우 "이미 처리됨 (멱등성 보장)" 메시지가 반환된다
- 동일 트랜잭션을 두 명이 동시에 재처리해도 한 번만 실행된다 (`SELECT FOR UPDATE`)
- 최대 3회 초과 시 `RETRY_LIMIT_EXCEEDED` 에러가 반환된다

### Interface Registry
- 인터페이스를 등록 · 수정 · 삭제 · 토글할 수 있다
- 비활성 인터페이스는 실행 버튼이 비활성화되고 API 호출 시 400 에러가 반환된다
- 실행 결과가 SSE를 통해 연결된 모든 클라이언트에 브로드캐스트된다

### 성능 관리
- P95가 3,000ms를 초과하는 인터페이스가 slow_alerts에 포함된다
- 기간 · 프로토콜 · 기관 필터 조합으로 집계 범위를 변경할 수 있다
