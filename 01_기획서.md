# [개발 포트폴리오 기획서]
# AI 기반 대내외 금융 인터페이스 통합 관제 및 재처리 시스템 (프로토타입)

---

## 1. 프로젝트 개요

**프로젝트명**: IF-Master (Interface Master) - 단일 화면 기반 통합 인터페이스 관리 플랫폼

**추진 배경**
보험사는 내부 핵심 원장 시스템과 외부 기관(금융감독원, 결제 대행사, 제휴 보험대리점 등) 간 수많은 인터페이스(REST, SOAP, Batch 등)를 운영합니다. 장애 발생 시 각 시스템의 로그를 개별적으로 확인해야 하므로 원인 파악 및 조치(재처리)가 지연되는 문제가 있습니다.

**개발 목표**
이종(Heterogeneous) 프로토콜 간의 연동 상태를 '단일 화면(Single Pane of Glass)'에서 실시간 모니터링하고, 실패한 트랜잭션을 즉각적으로 재처리할 수 있는 통합 관리 웹앱 프로토타입 구축.

---

## 2. 핵심 기능 정의 (MVP - 프로토타입 범위)

빠른 프로토타이핑을 위해 방대한 전체 시스템 중 가장 가치 있는 핵심 기능 3가지에 집중합니다.

### 기능 1. 통합 모니터링 대시보드 (단일 화면)
- 전체 인터페이스 처리 현황 (성공/실패/지연) 실시간 시각화 (도넛 차트, 시계열 그래프)
- 기관별(금감원, 은행 등), 프로토콜별(REST, Batch 등) 필터링 조회 기능

### 기능 2. 통합 로그 관리 및 상세 조회
- 분산된 인터페이스 호출 로그를 하나의 테이블에서 통합 조회
- Request/Response 데이터 전문(Payload) 및 에러 메시지(Stack Trace) 확인

### 기능 3. 원클릭 재처리(Retry) 기능 (비즈니스 로직 핵심)
- 타임아웃이나 네트워크 오류로 실패한 트랜잭션에 대해 담당자가 대시보드에서 즉시 재요청(Retry)을 실행할 수 있는 기능
- **멱등성(Idempotency) 보장**: 재처리 전 외부 기관의 실제 처리 여부를 조회하여 중복 처리를 방지. Idempotency Key를 활용해 동일 트랜잭션의 중복 전송을 원천 차단
- 재처리 실행자, 실행 시각, 결과를 Audit Log로 기록

### 기능 4. Mock 인터페이스 서버 구축 (테스트용)
- 실제 금융사망을 연결할 수 없으므로, 무작위로 성공/실패/지연 응답을 발생시키는 가상의 외부 기관(금감원, 은행) API 서버 구현

## 3. 시스템 아키텍처 및 기술 스택

가장 빠르고 효율적으로 구현할 수 있는 모던 스택을 선정합니다.

| 분류 | 기술 | 선정 사유 |
|------|------|-----------|
| Frontend | Next.js (React) + Tailwind CSS + Recharts | AI 코드 생성 최적화 조합, 빠른 UI 구축 |
| Backend | Python (FastAPI) | 금융 도메인 선호 언어, 비동기 REST API 고속 구축 |
| Database | PostgreSQL | 트랜잭션 보장, 금융 시스템 수준의 신뢰성 |
| AI 개발 툴 | Antigravity, Claude Code, Gemini 3.1 Pro | 기획-개발-테스트 전 과정 AI 협업 |

## 4. 바이브코딩(AI) 활용 계획 (포트폴리오 강조 포인트)

본 프로젝트는 기획-개발-테스트 전 과정에 AI를 적극 활용하는 '바이브코딩' 방법론을 적용합니다.
(제출 시 개발 문서에 아래 과정의 프롬프트를 캡처하여 첨부할 예정입니다.)

**[Mock Data 생성]** Gemini 3.1 Pro를 활용하여 "보험사-금감원 간의 SOAP 통신 실패 로그", "보험사-은행 간의 출금 REST API 성공 로그" 등 현실적인 금융 더미 데이터 JSON을 생성합니다.

**[프론트엔드 UI 구축]** Claude Code를 활용해 다크모드 기반 금융 시스템 관제 대시보드 UI 뼈대를 구축합니다.

**[백엔드 재처리 로직]** Antigravity를 활용하여 실패한 트랜잭션의 Payload를 재전송하고 결과를 DB에 업데이트하는 Python 코드를 자동 생성 및 최적화합니다.

**[멱등성 설계]** Claude Code를 활용하여 Idempotency Key 기반의 중복 방지 로직을 설계하고 구현합니다.

## 5. 화면 설계 (UI/UX Draft)

단일 화면(Single Page) 구조로 구성합니다.

```
┌─────────────────────────────────────────────────────┐
│ (Top) 요약 지표                                      │
│  총 트랜잭션 N건 | 성공률 N% | 🔴 장애 N건          │
├─────────────────────────────────────────────────────┤
│ (Middle) 트렌드 차트                                 │
│  시간대별 프로토콜(API/Batch) 호출량 및 에러 추이    │
├─────────────────────────────────────────────────────┤
│ (Bottom) 인터페이스 로그 그리드                      │
│  발생일시 | 대상기관 | 프로토콜 | 서비스명 | 상태 | [재처리] │
│                                                     │
│  → 행 클릭 시 우측 슬라이드 패널                    │
│     상세 송수신 데이터(JSON/XML) 노출               │
└─────────────────────────────────────────────────────┘
```

## 6. 기대 효과 (정성적)

- **운영 효율성 향상**: 장애 발생 시 팀별로 로그를 개별 조회하던 방식을 대시보드 단일 화면으로 단축 (MTTR 감소 기대)
- **데이터 정합성 확보**: 실패한 금융 거래를 누락 없이 식별하고 멱등성 보장 하에 재처리하여 대외 기관과의 데이터 불일치 리스크 해소
- **운영 감사 대응**: 재처리 이력(Audit Log) 보존으로 금융 규제 감사 대응 가능
